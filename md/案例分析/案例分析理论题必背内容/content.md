# 案例分析理论题必背内容

从历年考试情况来看，项目的立项、整合管理并不是案例分析考察的重点内容，但是也会考到，而且上午选择题、论文写作都可能会考，还是建议认真的学习。

# 第1章 立项管理

## 1.1 立项管理各方面可能存在的问题

1. 立项申请应由（建设方）甲方（非承建方，即乙方）的上级主管单位，而非甲方总经理批准。
2. 做完初步可行性研究就立马上马。
3. 未做详细可行性研究就生成可行性研究报告。
4. 可行性研究报告未经评审。
5. 仅根据项目符合国家政策就判断项目肯定要上马，判断依据过于单一。
6. 可行性分析的申请金额比申请下来的金额超过了上下 10% 的浮动，还继续使用，没有重写。
7. 可行性分析的申请金额比申请下来的金额没超过上下 10% 浮动，但没补充内容。
8. 投标由软件工程师负责不合适，缺少相关经验（专人专事）。
9. 仅从技术角度分析项目可行不全面，需要综合考虑经济、技术、社会等因素。
10. 投标文件不能单独完成，需要比较有经验的各领域专家共同参与编写。
11. 未编写项目建议书。
12. 未进行项目评估，或项目评估自己做。
13. 没有进行系统的可行性分析，没有进行多方案比较。
14. 调研不充分，没有调研大规模应用的案例。
15. 没有调研国家政策是否允许。
16. 未进行项目论证。

## 1.2 项目立项内容

项目立项一般包括项目建议与立项申请、项目可行性研究、项目评估与决策等内容。

## 1.3 项目建议书的内容（高级22年下）

1. 项目的必要性。
2. 项目的市场预测。
3. 项目预期成果（如产品方案或服务）的市场预测。
4. 项目建设必需的条件。

**项目建议书的作用**：项目建议书是项目发展周期的初始阶段，是国家或上级主管部门选择项目的依据，也是可行性研究的依据。

## 1.4 信息系统项目的可行性研究包含（高级08年下）

1. 技术可行性分析。
2. 经济可行性分析。
3. 社会效益可行性分析。
4. 运行环境可行性分析。
5. 其他方面的可行性分析等。

> **【口诀】** 可行性研究包含：技术、经济、社会效益、运行环境、其他可行性分析。

## 1.5 项目评估的依据

主要包括：
1. 项目建议书及其批准文件。
2. 项目可行性研究报告。
3. 报送组织的申请报告及主管部门的初审意见。
4. 项目关键建设条件和工程等的协议文件。
5. 必需的其他文件和资料等。

## 1.6 项目评估程序

成立评估小组 $\rightarrow$ 开展调查研究 $\rightarrow$ 分析与评估 $\rightarrow$ 编写、讨论、修改评估报告 $\rightarrow$ 召开专家论证会 $\rightarrow$ 评估报告定稿并发布。

# 第2章 整合管理

## 2.1 整合管理方面可能存在的问题

### 2.1.1 制定项目章程
1. 没有写项目章程，没有颁布。
2. 项目经理自己颁布项目章程（应由发起人发布）。
3. 项目经理修改项目章程（项目经理无权修改）。
4. 项目章程授权不够，项目经理没有权限，下面的人不听话。
5. 项目章程的内容不完整（关键字：包含，内容有）。

### 2.1.2 制定项目管理计划
1. 项目管理计划没写，直接做事。
2. 项目管理计划项目经理一个人写。
3. 项目管理的计划内容不完整（关键字：包含，内容有）。
4. 只有初步的项目管理计划。
5. 项目管理计划没有评审。
6. 项目经理叫组长写完各分计划后，只是汇总项目管理计划，没有进行整合、协调。
7. 项目已经变更，计划未更新。
8. 没做好各子计划的统一协调，可能导致项目计划不符合项目实际情况。
9. 项目管理计划制定比较简单，不足以支持整个项目对所需过程的指导和管理。
10. 项目计划缺少相关分计划，如质量计划、沟通计划等。
11. 选择的软件开发生命周期模型不适合项目。

### 2.1.3 指导与管理项目工作
1. 执行过于随意，没按计划进行。
2. 公司缺乏对项目的指导和监控。
3. 项目经理没人支持，感觉迷茫，缺少 PMO。

### 2.1.4 管理项目知识
1. 没有管理项目知识。
2. 没有形成经验教训登记册。
3. 没有合理运用相关的工具和技术。

### 2.1.5 监控项目工作和实施整体变更控制
1. 没进行项目监控。
2. 没走变更控制程序。
3. 变更控制程序缺少部分的步骤。
4. 缺少 CCB。
5. 项目成员不走变更流程，直接修改项目工作内容。
6. 变更发生之后，不作处理。（项目经理没尽责）
7. 监控不力（直到...时才发现）。

### 2.1.6 结束项目或阶段
1. 没做好收尾工作（项目一结束立即就解散团队）。
2. 没有甲方的同意、签字，项目经理就进行收尾工作。
3. 没有验收报告，甲方没有在验收报告上签名。
4. 收尾的时候没有做经验总结。

## 2.2 项目章程的内容（高级10上、15上、16上）

项目章程记录了关于项目和项目预期交付的产品、服务或成果的高层级信息：

1. 项目目的。
2. 可测量的项目目标和相关的成功标准。
3. 高层级需求、高层级项目描述、边界定义以及主要可交付成果。
4. 整体项目风险。
5. 总体里程碑进度计划。
6. 预先批准的财务资源。
7. 关键干系人名单。
8. 项目审批要求（例如，评价项目成功的标准，由谁对项目成功下结论，由谁签署项目结束）。
9. 项目退出标准（例如，在何种条件下才能关闭或取消项目或阶段）。
10. 委派的项目经理及其职责和职权。
11. 发起人或其他批准项目章程的人员的姓名和职权等。

> **【口诀】** 目的—目标—需求要描述；风险—进度—资源—名单要审批，退出—两个职位。

## 2.3 项目章程的作用

1. 明确项目与组织战略目标之间的直接联系。
2. 确立项目的正式地位。
3. 展示组织对项目的承诺。

## 2.4 项目管理计划组件

主要包括：
- **子管理计划**：范围管理计划、需求管理计划、进度管理计划、成本管理计划、质量管理计划、资源管理计划、沟通管理计划、风险管理计划、采购管理计划、干系人参与计划。
- **基准**：范围基准、进度基准和成本基准。
- **其他组件**：变更管理计划、配置管理计划、绩效测量基准、项目生命周期、开发方法、管理审查。

## 2.5 整合管理中涉及的配置管理活动
识别配置项、记录并报告配置项状态、进行配置项核实与审计。

## 2.6 整合管理中涉及的变更管理活动
识别变更、记录变更、做出变更决定、跟踪变更。

## 2.7 变更请求的内容
纠正措施、预防措施、缺陷补救、更新。

## 2.8 结束项目或阶段过程所需执行的活动
- 为达到阶段或项目的完工或退出标准所必须的行动和活动。
- 为关闭项目合同协议或项目阶段合同协议所必须开展的活动。
- 为完成收集项目或阶段记录、审计项目成败、管理知识分享和传递、总结经验教训、存档项目信息以供组织未来使用等工作所必须开展的活动。
- 为向下一个阶段，或者向生产和（或）运营部门移交项目的产品、服务或成果所必须开展的行动和活动。
- 收集关于改进或更新组织政策和程序的建议，并将它们发送给相应的组织部门。
- 测量干系人的满意程度等。

## 2.9 整合管理7个过程/归属5大过程组

1. 制定项目章程【启动】
2. 制定项目管理计划【计划】
3. 指导和管理项目工作【执行】
4. 管理项目知识【执行】
5. 监控项目工作【监控】
6. 实施整体变更控制【监控】
7. 结束项目或阶段【收尾】

## 2.10 项目整合管理的目标（高级23年第4批）
1. 资源分配。
2. 平衡竞争性需求。
3. 研究各种备选方法。
4. 裁剪过程以实现项目目标。
5. 管理各个项目管理知识领域之间的依赖关系。

# 第3章 范围管理

## 3.1 范围管理过程中存在的问题

### 3.1.1 规划范围管理
1. 没编制范围管理计划和需求管理计划。
2. 范围管理计划、需求管理计划是 1 个人编写的。
3. 没有进行范围管理计划的评审。
4. 缺乏项目范围管理的思想。

### 3.1.2 收集需求
1. 没做需求收集工作。
2. 没召开需求评审会，没确认需求，没有形成需求文件和需求跟踪矩阵。
3. 没有与各干系人对需求进行详细分析，只是在对客户需求的初步了解后就开始实施。
4. 对需求估计不准确，资源估算不足。
5. 需求评审没有客户参与，可能导致最终对需求不能达成一致；设计文件没有经过正式评审，可能导致设计文件有较多的错误。
6. 对客户（或用户）的需求获取不充分。

### 3.1.3 定义范围
1. 没进行范围定义。
2. 范围说明书内容不全。
3. 不能只参照类似项目的范围说明书，需要根据本项目情况进行编写。
4. 范围说明书没经过评审。
5. 一个人编写了范围说明书。
6. 闭门造车式地开展需求调研与项目范围说明书的编写工作，没让相关干系人参与进来。

### 3.1.4 创建 WBS
1. 没进行创建 WBS。
2. 不能单独一人对项目进行分解，而要让项目组成员也参与进来。
3. WBS 没有经过相关干系人的确认。
4. 没有编制 WBS 和 WBS 词典，以形成权威的范围基准。
5. 一个工作只能由 1 个人负责。
6. 工作包的大小应该介于 8/80 之间。
7. 没包含外包出去的模块。
8. 没包含项目管理工作。
9. 一个下层属于多个上层了，有交叉从属。
10. WBS 最好可以分解到 4-6 层。

### 3.1.5 确认范围
1. 没做范围确认。
2. 范围管理没做好，导致范围出现蔓延。
3. 范围确认存在问题，导致 WBS 中定义的功能没有开发。
4. 范围确认发现的问题没有及时有效的解决。

### 3.1.6 控制范围
1. 没做好范围控制。
2. 存在镀金、范围蔓延行为。
3. 对于范围变更没有走变更控制流程。

## 3.2 需求管理可能的问题
1. 对客户（或用户）的需求获取不充分。
2. 没有按照规范的需求开发与需求管理的流程及内容开展需求工作。
3. 缺乏需求定义环节，没有定义出需求规格说明书。
4. 缺乏需求验证环节，没有请客户代表一起进行需求评审。
5. 没有制定范围和需求管理计划。
6. 没有求得干系人对需求的一致理解。
7. 没有求得干系人对需求的承诺。
8. 没有有效地管理需求变更控制。
9. 没有有效维护对需求进行跟踪管理。
10. 没有及时识别项目工作与需求之间的不一致性。

## 3.3 需求管理问题应对措施
1. 建立需求变更控制策略和需求变更控制流程。
2. 采用多种方式充分获取用户需求并进行详细的需求分析。
3. 形成需求规格说明书并与用户进行需求验证（确认）和评审。
4. 需求定稿建立基线，以后的需求变更必须走变更控制流程，并及时更新需求规格说明书和需求跟踪矩阵。
5. 编写需求跟踪矩阵，需求状态表等文档。
6. 在需求规格说明书基础上编制技术方案，并进行评审。
7. 定期不定期对项目绩效进行监督检查，找出问题原因并指导团队成员解决。

## 3.4 范围管理计划的内容
1. 制定项目范围说明书。
2. 根据详细项目范围说明书创建 WBS。
3. 确定如何审批和维护范围基准。
4. 正式验收已完成的项目可交付成果。

## 3.5 需求管理计划的内容
1. 如何规划、跟踪和报告各种需求活动。
2. 配置管理活动，例如，如何启动变更，如何分析其影响，如何进行追溯、跟踪和报告，以及变更审批权限。
3. 需求优先级排序过程。
4. 测量指标及使用这些指标的理由。
5. 反映哪些需求属性将被列入跟踪矩阵等。

## 3.6 范围说明书内容（高级20年下）
1. 产品范围描述。
2. 验收标准。
3. 可交付成果。
4. 项目的除外责任。
5. 制约因素。
6. 假设条件。

> **【口诀】** 产验可除制假

## 3.7 WBS 分解的方法
1. 以项目生命周期各阶段作为分解的第二层，产品和项目可交付成果放在第三层。
2. 以主要可交付成果作为分解的第二层。
3. 纳入由项目团队以外的组织开发的各种较低层次组件（如外包工作）。随后，作为外包工作的一部分，卖方须制定相应的合同 WBS。

## 3.8 WBS 表示形式
- **树型结构图（组织结构图式）**：层次清晰、直观性和结构性强，但不容易修改，对大的、复杂的项目很难表示出项目的全貌。
- **表格形式（列表式）**：直观性比较差，但能够反映出项目所有的工作要素。

## 3.9 敏捷或适应型方法的 WBS 形式
提纲式、组织结构图或能说明层级结构的其他形式。

## 3.10 分解工作结构应把握的原则
1. WBS 必须是面向可交付成果的。
2. WBS 必须符合项目的范围。
3. WBS 的底层应该支持计划和控制。
4. WBS 中的元素必须有人负责，而且只由一个人负责，尽管实际上可能需要多个人参与。
5. WBS 的指导，WBS 应控制在 4～6 层。
6. WBS 应包括项目管理工作（因为管理是项目具体工作的一部分），也要包括分包出去的工作。
7. WBS 的编制需要所有（主要）项目干系人的参与，需要项目团队成员的参与。
8. WBS 并非是一成不变的。在完成了 WBS 之后的工作中，仍然有可能需要对 WBS 进行修改。

> **【口诀】** 一个人负责，4-6层，面果，盒饭，可变，全员，吃鸡，管饱

## 3.11 需求跟踪矩阵记录的典型属性
唯一标识、需求的文字描述、收录该需求的理由、所有者、来源、优先级别、版本、当前状态和状态日期。为确保干系人满意，可能需要增加一些补充属性，如稳定性、复杂性和验收标准。

## 3.12 跟踪需求的内容
业务需要、机会、目的和目标；项目目标；项目范围和 WBS 可交付成果；产品设计；产品开发；测试策略和测试场景；高层级需求到详细需求等。

## 3.13 需求的类别
业务需求、干系人需求、解决方案需求、过渡和就绪需求、项目需求、质量需求。

## 3.14 范围基准与衡量
项目的范围基准是经过批准的项目范围说明书、WBS 和 WBS 词典。项目范围的完成情况是根据项目管理计划来衡量的。产品范围的完成情况是根据产品需求来衡量的。（高级21年下）

## 3.15 WBS 分解步骤（高级18年下）
1. 识别和分析可交付成果及相关工作。
2. 确定 WBS 的结构和编排方法。
3. 自上而下逐层细化分解。
4. 为 WBS 组件制定和分配标识编码。
5. 核实可交付成果分解的程度是恰当的。

> **【口诀】** 识果、确编、上细、分码、度当

## 3.16 范围确认和质量控制的区别和联系
- **联系**：控制质量过程通常先于确认范围过程，但二者也可同时进行。
- **区别**：范围确认关注可交付成果的**验收**，质量控制关注可交付成果的**正确性**及是否满足质量要求。质量控制是内部完成的，范围确认需要甲方参与。

## 3.17 范围确认的步骤
1. 确定需要进行范围确认的时间。
2. 识别范围确认需要哪些投入。
3. 确定范围正式被接受的标准和要素。
4. 确定范围确认会议的组织步骤。
5. 组织范围确认会议。

## 3.18 干系人范围确认需要检查的方面
1. 可交付成果是否是确定的、可确认的。
2. 每个可交付成果是否有明确的里程碑及验收标准。
3. 是否有明确的质量标准。
4. 审核和承诺是否有清晰的表达（边界是否清晰）。
5. 项目范围是否覆盖了需要完成的产品或服务的所有活动，有没有遗漏或错误。
6. 项目范围的风险是否太高。

## 3.19 作为基准的条件
只有明确的（可测量和可测试的）、可跟踪的、完整的、相互协调的，且主要干系人愿意认可的需求，才能作为基准。

## 3.20 各个干系人关注的重点
- **管理层**：关注项目范围对进度、资金和资源的影响（投入产出合理性）。
- **客户**：关注产品范围（功能和特征是否满足）。
- **项目管理人员**：关注项目制约因素（可交付成果是否足够，资源是否足够）。
- **项目团队人员**：关注自己参与和负责的元素（工作时间、工作量）。

# 第4章 进度管理

## 4.1 进度管理中可能出现的问题

### 4.1.1 规划进度管理
1. 没进行规划进度管理。
2. 不能一人来制定进度计划，并且没有从项目实际出发来制定进度计划，根据合同规定时间制定的计划可能不符合实际。

### 4.1.2 定义活动
1. 没进行活动定义。

### 4.1.3 排列活动顺序
1. 没进行活动排序。

### 4.1.4 估算活动持续时间
1. 没进行历时估算。
2. 历时估算不准确。

### 4.1.5 制定进度计划
1. 没制定进度计划。
2. 加班会增加成本，影响质量。
3. 并行工作会增加风险。
4. 增加资源有时可能压缩工期有限。
5. 制定进度计划的方法不合理，没有预留一定的缓冲时间。
6. 项目进度计划不合理。
7. 计划未经过评审就付诸实施。
8. 制定工作计划时，没考虑资源日历，导致有冲突。
9. 在压缩工期的情况下，没有考虑新增加开发人员的可用性。
10. 未经过评估情况下随意将原来系统开发时间压缩。
11. 关键里程碑点没有获得相关干系人的签字确认。

### 4.1.6 控制进度
1. 没做进度控制。
2. 控制进度的工作做得不好。
3. 缺乏进度管理的思想。

## 4.2 项目进度出现问题的解决方案
1. 向公司申请增加资源，或使用经验丰富的员工。
2. 优化网络图，重排活动之间的顺序，压缩关键路径的长度。
3. 临时加班（赶工），尽可能补救耽误的时间或提升资源的利用效率。
4. 将部分阶段的工作改为并行、内部流程优化。
5. 变更原来的进度计划，根据前阶段的绩效，对后续工作重新估计，修订计划，并得到项目干系人的同意。
6. 加强同项目干系人的沟通。
7. 加强对交付物、项目阶段工作的及时检查和控制，避免后期出现返工。
8. 尽可能调配非关键路径上的资源用于关键路径上的任务。
9. 优化外包、采购等环节并全程监控。

## 4.3 缩短活动的工期（措施）
1. **赶工**：投入更多的资源或增加工作时间，以缩短关键活动的工期。
2. **快速跟进**：并行施工，以缩短关键路径的长度。
3. 使用高素质的资源或经验更丰富的人员。
4. 减小活动范围或降低活动要求。
5. 改进方法或技术，以提高生产效率。
6. 加强质量管理，及时发现问题，减少返工，从而缩短工期。

> **【口诀】** 赶快使减改为加（赶工，并行施工，高素质经验丰富人员，高效人员代替低效人员，减小活动范围或降低要求，改进方法，减少返工，提高效率，提高资源利用率，增强优质资源，外包）。

## 4.4 进度压缩技术及其利弊

- **赶工**：通过增加资源，以最小的成本代价来压缩进度工期。
  - *例子*：批准加班、增加额外资源或支付加急费用。
  - *利弊*：只适用于通过增加资源就能缩短持续时间且位于关键路径上的活动。可能导致风险和/或成本增加。
- **快速跟进**：将正常情况下按顺序进行的活动或阶段改为至少是部分并行开展。
  - *例子*：图纸未完先建地基。
  - *利弊*：可能造成返工和风险增加。只适用于能通过并行活动缩短关键路径的情况。可能增加协调工作、质量风险和项目成本。

## 4.5 确定依赖关系
1. **强制性依赖关系**：法律、合同或工作内在性质决定的。
2. **选择性依赖关系**：首选、优先或软逻辑关系。
3. **外部依赖关系**：项目活动与非项目活动之间的依赖关系（如等待外部设备到货）。
4. **内部依赖关系**：项目活动之间的紧前关系，通常在团队控制之中（如先组装后测试）。

## 4.6 资源优化技术
1. **资源平衡**：在资源需求与供给之间取得平衡。如果共享/关键资源有限，需调整开始/完成日期。往往导致**关键路径改变**。
2. **资源平滑**：利用自由浮动和总浮动时间调整活动，使资源需求不超过预定限制。**不会改变**关键路径，完工日期不延迟。

## 4.7 进度计划的种类和用途
1. **里程碑进度计划**（里程碑图）：高层级。
2. **概括性进度计划**（横道图/甘特图）：向管理层汇报。
3. **详细进度计划**（项目进度网络图）：团队内部控制。

## 4.8 进度落后，成本超支的措施（重点）
1. 用高效人员代替低效人员。
2. 加班或者赶工（注意防范风险）。
3. 在防范风险的情况下并行施工（快速跟进）。
4. 减小活动范围或降低活动要求。
5. 通过改进方法或技术提高生产效率。
6. 分析成本超支原因，找出针对性对策。

## 4.9 进度落后，成本节约的措施
1. 赶工（如全体加班）加快进度。
2. 使用高效资源来替换低效资源加快进度。
3. 改进方法，提高工作效率。

## 4.10 进度超前，成本超支的措施
1. 抽出部分人员以放慢工作进度。
2. 采取控制成本措施。
3. 区分不同任务，采取不同措施。
4. 必要时调整成本基准。
5. 优化施工方案、提高效率、加强质量管理减少返工。
6. 在确保进度按期完成基础上，降低进度以节约成本。
7. 总结经验并推广。

## 4.11 进度超前，成本节约的措施
1. 抽调部分人员用于其他项目。
2. 加强质量控制，密切监控项目。
3. 必要时调整计划或基准。

## 4.13 估算持续时间需考虑的因素
1. **收益递减规律**：增加投入达到临界点后，产出递减。
2. **资源数量**：资源加倍不一定时间减半，可能增加风险。
3. **技术进步**：新技术可能影响持续时间和资源。
4. **员工激励**：拖延症（学生综合征）和帕金森定律（工作会自动膨胀占满所有可用时间）。

# 第5章 成本管理

## 5.1 成本管理各方面可能存在的问题

### 5.1.1 规划成本管理
1. 没进行成本规划。
2. 1 个人编写了成本管理计划。
3. 成本管理计划没经过评审。
4. 成本管理计划内容不全。

### 5.1.2 估算成本
1. 没进行成本估算。
2. 成本估算不准确。

### 5.1.3 制定预算
1. 没进行成本预算。
2. 成本预算不准确。

### 5.1.4 控制成本
1. 没进行成本控制。
2. 没采用相关工具进行成本控制。
3. 赶工导致成本超支了。

## 5.2 项目成本控制的目标（高级17年上）
1. 对造成成本基准变更的因素施加影响。
2. 确保所有变更请求都得到及时处理。
3. 当变更实际发生时，管理这些变更。
4. 确保成本支出不超过批准的资金限额（按时段、按 WBS 组件、按活动、按总限额）。
5. 监督成本绩效，找出并分析与成本基准间的偏差。
6. 对照资金支出，监督工作绩效。
7. 防止在成本或资源使用报告中出现未经批准的变更。
8. 向有关干系人报告所有经批准的变更及其相关成本。
9. 设法把预期的成本超支控制在可接受的范围内。

> **【口诀】** 可接受五变更监督偏差不超额

## 5.3 挣值分析
（详见计算题专题）
*注：管理储备不包括在成本基准中，但属于项目总预算和资金需求的一部分。不计入挣值计算。*

## 5.4 成本的类型
1. **可变成本**：随生产量/工作量/时间变化（如原材料）。
2. **固定成本**：不随变化而变化（如设备折旧）。
3. **直接成本**：直接归属于项目工作的成本（如差旅、工资）。
4. **间接成本**：分摊给本项目的费用（如税金、水电、保卫）。
5. **机会成本**：选择一种而放弃另一种的最佳替代品收益。
6. **沉没成本**：过去已发生，不能改变的成本（决策时应排除干扰）。

## 5.5 储备分析
- **应急储备**：包含在成本基准内，应对“已知-未知”风险。
- **管理储备**：项目总预算的一部分，不含在基准内，应对“未知-未知”风险。使用需高层审批。

## 5.6 成本超支应对措施
1. 关注超支严重的工作。
2. 细化分析，找出原因。
3. 采取措施：减少不必要工作、优化流程、削减不必要资源。
4. 定期评估成本绩效，及时调整。

3. 用人错误（QA 无经验）。
4. QA 发现问题未与当事人协商，或自作主张。

6. 经理认为自己只是配合角色。

1. 没做质量控制。
2. 缺失评审和测试环节。
3. 测试方法不当/不充分。
4. 流程错误（未控制就确认范围）。
5. 未在过程中检查，事后才查。
6. 检查走过场。
7. 测试用例/环境等有问题。
8. 阶段安排不合理，测试时间不足。
9. 修改代码后未回归测试。
10. 检查颗粒度不一。

## 6.2 质量问题解决措施
1. 严格执行公司质量体系。
2. 制定并执行质量管理计划/保证计划。
3. 调配资源加强质量工作。
4. 安排独立测试人员加强控制。
5. 加强售后维护。
6. 加强沟通。
7. 必要时修改基准。
8. 参与过程描述评审。
9. 按计划实施检查，记录情况。
10. 识别偏差，分析原因，协商解决。
11. 定期分发质量报告。
12. 协调变更。
13. 提供培训。
14. 强有力的领导。
15. 建立组织级体系/激励制度。
16. 理解质量成本。

## 6.3 管理质量与质量控制的区别（补充）
- **管理质量**：针对过程改进和审计，强调过程改进和信心保证。
- **质量控制**：针对可交付成果，强调正确性。
- **联系**：都贯穿始终，都为保证质量，管理质量为控制提供条件，控制结果是管理的输入。

## 6.4 确认范围与质量控制的区别（补充）
- **确认范围**：强调**接受**（外部干系人验收）。
- **质量控制**：强调**正确**（内部检查）。
- **顺序**：一般先控制质量（或同时），后确认范围。

## 6.5 质量管理计划内容
1. 质量标准。
2. 质量目标。
3. 角色与职责。
4. 需要审查的可交付成果和过程。
5. 规划的控制和管理活动。
6. 质量工具。
7. 主要程序。

## 6.6 质量成本（高级19年下）

| 类别 | 子类别 | 内容 | 目的/性质 |
| :--- | :--- | :--- | :--- |
| **一致性成本** | **预防成本** | 培训、文件过程、设备、完成时间 | 打造高质量产品，规避失败 |
| | **评估成本** | 测试、破坏性试验损失、检查 | 评估质量 |
| **不一致成本** | **内部失败成本** | 返工、报废 | 项目中发现的失败 |
| | **外部失败成本** | 债务、保修工作、失去业务 | 客户发现的失败（项目前后花费的资金） |

## 6.7 数据表现技术
- **规划质量**：流程图、逻辑数据模型、矩阵图、思维导图。
- **管理质量**：亲和图、因果图、流程图、直方图、矩阵图、散点图。
- **质量控制**：因果图、控制图、直方图、散点图。

## 6.8 质量与等级
- **质量**：满足要求的程度。
- **等级**：技术特性的分类。
- *低等级、高质量*（功能少但好用）通常可接受；*高等级、低质量*（功能多但老坏）不可接受。

## 6.9 质量审计目标
识别最佳实践、违规做法、分享好实践、协助改进过程、贡献经验教训。

## 6.10 帕累托图（二八法则）
80% 的问题是 20% 的原因造成的。

## 6.11 质量保证人员 QA 的职责
过程指导、过程评审、产品审计、过程改进、过程度量。包括建立指标、参与规划评审、编制 QA 计划、过程/交付物审计、纠偏、培训等。

## 7.4 虚拟团队利弊
- **优点**：利用不同地理位置人才、增加特殊技能、在家办公、不同班次、纳入行动不便者、节省差旅/设施费用。
- **缺点**：误解、孤立感、分享困难、通信成本。

## 7.5 团队建设 5 阶段
形成 → 震荡 → 规范 → 发挥 → 解散

## 7.6 冲突解决方法（重要）
1. **撤退/回避**：退出或推迟。
2. **缓和/包容**：强调一致，退让。
3. **妥协/调解**：各让一步（双输可能）。
4. **强迫/命令**：利用权力，牺牲一方（赢-输）。
5. **合作/解决问题**：开放对话，达成共识（双赢）。

> **【口诀】** 撤回 缓包 协调 迫命 合解

## 7.7 激励理论
- **X 理论**（人性本恶）：好逸恶劳，需要强制、惩罚、金钱激励。
- **Y 理论**（人性本善）：热爱工作，能自我控制，有创造力，渴望发挥。
- **马斯洛需求**：生理 → 安全 → 社交 → 尊重 → 自我实现。

## 7.8 权力的 5 种来源
职位（法定）、惩罚（强制）、奖励、专家、参照。

> **【口诀】** 职位，惩罚，奖励，专家，参照

# 第8章 沟通管理

## 8.1 沟通管理可能问题

### 8.1.1 规划沟通管理
1. 没进行规划。
2. 计划由一人制定。
3. 内容不全。
4. 未经干系人确认/评审。
5. 脱离实际，照搬模板。
6. 经验不足。

### 8.1.2 管理沟通
1. 没做管理。
2. 极少与客户直接沟通。
3. 未分析成员需求/风格。
4. 方式单一。
5. 沟通不及时。
6. 无记录。
7. 导致客户不满/投诉。
8. 只用邮件。
9. 公开场合谈论私下诉求。
10. 周报/例会问题（粒度粗、内容全）。
11. 客户召集会议（应由项目方主动）。
12. 临时通知参会/无议题/效率低。

### 8.1.3 监督沟通
1. 没做监督。
2. 问题未解决。
3. 发生争执。
4. 强迫中止诉求。
5. 未获高层认同。
6. 甲方内部需求未统一。
7. 缺乏统一接口。
8. 协作不够。
9. 技巧不足。
10. 变更无记录/无监控。

## 8.2 沟通管理计划主要内容
干系人需求、需沟通信息、上报步骤、发布原因、时限频率、负责人、授权人、接收人、方法技术、资源预算、更新方法、术语表、流程图、制约因素。

## 8.3 沟通方法
1. **互动沟通**：实时多向（会议、电话）。
2. **推式沟通**：发送给特定接收方（邮件、报告、备忘录）。
3. **拉式沟通**：自行访问（网站、知识库）。

## 8.4 书面沟通的 5C 原则
正确的语法和拼写、简洁的表述、清晰的目的和表述、连贯的思维逻辑、善用控制语句和承接。

# 第9章 干系人管理

## 9.1 常见问题

## 9.2 权力/利益方格

| 区域 | 特征 | 管理策略 |

| **A区** | 权力高、利益低 | **令其满意** |
| **B区** | 权力高、利益高 | **重点管理**（及时报告） |
| **C区** | 权力低、利益高 | **随时告知** |
| **D区** | 权力低、利益低 | **监督**（最少精力） |

# 第10章 风险管理

## 10.1 常见问题
1. **规划**：未规划、独自制定、照搬模板、未讨论直接下发。
2. **识别**：未识别、凭个人经验、不详细、不全面、缺乏意识。
3. **分析**：未做定性/定量分析、方法不合理。
4. **应对**：未做、凭经验制定、未责任到人、措施不合理。
5. **监督**：未监督、未处理好风险、监控不力、未再识别、未审计。

## 10.2 风险应对措施

- **消极风险（威胁）**：上报、规避、转移、减轻、接受。
- **积极风险（机会）**：上报、开拓、分享、提高、接受。

**整体风险应对策略表：**

| 策略 | 解释说明 |
| :--- | :--- |
| **规避** | 弱化不确定性负面影响，拉回临界值内（如取消高风险工作）。 |
| **开拓** | 获得正面影响（如增加高收益工作）。 |
| **转移或分享** | 第三方管理。负面->转移（买保险、外包）；正面->分享（合资）。 |
| **减轻或提高** | 优化可能性/影响。减轻负面，提高正面（如调整资源、改变范围）。 |
| **接受** | 主动接受（建立储备）；被动接受（记录，定期复查）。 |

## 10.3 风险分类（按可预测性）
1. **已知风险**：后果可预见。
2. **可预测风险**：可预见发生，不可预见后果。
3. **不可预测风险**（未知风险）：完全无法预见。

## 10.4 SWOT
优势 (Strengths)、劣势 (Weaknesses)、机会 (Opportunities)、威胁 (Threats)。

# 第11章 采购管理

## 11.1 常见问题
1. **规划**：未规划、未做自制/外购分析。
2. **实施**：片面相信推荐、未发挥职责、推卸责任、获取方式有问题。
3. **控制**：验收问题、监督不力、文档混乱、库房管理问题。

## 11.2 采购步骤
SOW/TOR → 预算 → 广告 → 合格名单 → 发标 → 接标 → 评估（技术/成本） → 选定 → 签约。

## 11.3 供方选择分析方法
最低成本、仅凭资质、基于质量或技术方案得分、基于质量和成本、唯一来源、固定预算。

## 11.4 招投标流程
发布公告 → 踏勘 → 投标 → 开标 → 评标 → 确定中标人 → 订立合同。

# 第12章 合同管理

## 12.1 常见问题
1. 调查不足，草率签约。
2. 条款不明，缺乏细节。
3. 理解不一致。
4. 缺乏纠纷处理条款。
5. 总价合同风险认识不足。
6. 关键问题（工期质量）未约束。
7. 缺少需求描述及违约责任。
8. 档案管理不规范。
9. 缺变更流程。
10. 维护责任/履行地不明。
11. 验收标准问题。
12. 缺乏意识。

## 12.2 合同类型选择
1. **总价合同**：工作范围明确，设计详细。
   - FFP（固定总价）、FPIF（总价加激励）、FPEPA（总价加经济价格调整）。
2. **工料合同**：性质清楚但范围不清，不复杂，需快速签约；双方分担风险。
3. **成本补偿合同**：范围不清楚；买方承担成本风险。
   - CPFF（成本加固定费用）、CPIF（成本加激励）、CPAF（成本加奖励）。

## 12.3 订立分包合同条件
1. 买方认可。
2. 非主体工作。
3. 只能分包部分。
4. 具备资质。
5. 不能再次分包。

# 第13章 配置管理

## 13.1 常见问题
1. 未做计划/识别/控制/报告/审计/发布。
2. 验证缺失。
3. 变更未分析/未批准。
4. 版本管理混乱。
5. 兼职管理（精力不够）。
6. CCB 不合理。
7. 无基线。
8. 工具未评估。
9. 权限管理混乱（直接修改受控库）。
10. 删除配置项（错误）。
11. 无法追溯。

## 13.2 应对措施
1. 组建委员会，培训人员。
2. 引入工具。
3. 制定计划和流程。
4. 严格权限。
5. 建立基线和版本管理。
6. 定期报告。

## 13.3 配置库类型
1. **开发库**（动态库）：开发人员个人工作区，随意修改。
2. **受控库**（主库）：基线+变更，阶段性产物，修改需走流程。
3. **产品库**（静态库）：已发布产品，最终交付，原则上不修改。

> **【口诀】** 开发库（动态库），受控库（主库），产品库（静态库）。

## 13.4 软件升级流程
1. 产品库 $\rightarrow$ 受控库。
2. 受控库 Check out（锁定） $\rightarrow$ 开发库（修改）。
3. 开发库 Check in（解锁） $\rightarrow$ 受控库。
4. 受控库 $\rightarrow$ 产品库（新基线）。

## 13.5 配置审计
1. **功能配置审计**：一致性（实际功效是否符合需求）。
2. **物理配置审计**：完整性（物理存在是否一致，是否包含所有必需项）。

> **【口诀】** （双重）确认，防止发现并找出不好的。

决策机构，由多方人员（客户、实施方决策者）组成。决定基准是否变更，**不提供方案**。

# 第15章 项目管理概论

## 15.1 PMO 类型
- **支持型**：顾问，提供模板/培训，资源库。控制程度：**低**。
- **控制型**：要求服从（方法论、模板、治理）。控制程度：**中**。
- **指令型**：直接管理控制，PM 向其汇报。控制程度：**高**。

## 15.2 领导力风格
- **放任型**：无为而治。
- **交易型**：基于奖励。
- **服务型**：服务优先，关注他人成长。
- **变革型**：鼓舞激励，促进创新。
- **魅力型**：自信，说服力强。
- **交互型**：结合交易、变革、魅力。

# 第17章 其他内容

## 17.1 招投标时间关键点

| 时间要求 | 内容 |
| :--- | :--- |
| **5日** | 资格预审文件/招标文件发售期不得少于 5 日。 |
| **少于3个** | 资格预审申请人/投标人少于 3 个，重新招标。 |
| **5人以上单数, 2/3** | 评标委员会人数；技术经济专家不少于 2/3。 |
| **3人以上单数, 2/3** | 竞争性谈判/询价小组人数；专家不少于 2/3。 |
| **不超过3个** | 中标候选人数量。 |
| **3日** | 公示中标候选人期不得少于 3 日（收到报告之日起 3 日内公示）。 |
| **3日** | 收到异议之日起 3 日内答复。 |
| **5日** | 退还投标保证金（合同签订后 5 日内）。 |
| **2%** | 投标保证金不得超过估算价 2%。 |
| **10%** | 履约保证金不得超过合同金额 10%。 |
| **15日** | 澄清/修改招标文件（截止时间至少 15 日前）。 |
| **20日** | 发出招标文件至截止时间，最短不得少于 20 日。 |
| **30日** | 发出中标通知书起 30 日内签合同。 |

11. 未按排名确定中标人。
12. 签订合同超期。
13. 违规分包（关键/主体工作不可分包）。
