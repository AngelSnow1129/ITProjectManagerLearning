## 19.1 配置管理

**定义**：配置管理是为了系统地控制配置变更，在信息系统项目的整个生命周期中维持配置的完整性和可跟踪性，而标识信息系统建设在不同时间点上配置的学科。

### 19.1.1 管理基础

#### 1. 配置项 (Configuration Item, CI)
配置项是信息系统组件或与其有关的项目，包括软件、硬件和各种文档。
*   **范围**：变更请求、服务、服务器、环境、设备、网络设施、台式机、移动设备、应用系统、协议、电信服务等。
*   **典型示例**：项目计划书、技术解决方案、需求文档、设计文档、源代码、可执行代码、测试用例、运行软件所需的各种数据、设备型号及其关键部件等。

**分类**：
*   **基线配置项**：可能包括所有的设计文档和源程序等（通常向开发人员开放读取权限）。
*   **非基线配置项**：可能包括项目的各类计划和报告等（通常向项目经理、CCB及相关人员开放）。

**权限管理**：所有配置项的操作权限应由配置管理员（CMO）严格管理。

#### 2. 配置项状态
配置项的状态主要分为三种：
*   **草稿 (Draft)**：配置项刚建立时。
*   **正式 (Formal)**：配置项通过评审后。
*   **修改 (Modifying)**：若更改配置项，状态变为“修改”；修改完毕并重新通过评审后，状态变回“正式”。

![](./images/第19章_配置与变更管理/pic_004.jpg)
*图19-1 配置项状态变化*

#### 3. 配置项的版本号
*   **草稿状态**：版本号格式为 `0.YZ`。
    *   YZ范围：$01 \sim 99$。
    *   随着修正递增，例如：0.5、0.91。
*   **正式状态**：版本号格式为 `X.Y`。
    *   X为主版本号 ($1 \sim 9$)，Y为次版本号 ($0 \sim 9$)。
    *   第一次成为“正式”文件时，版本号为 `1.0`。
    *   小升级：`1.1`、`1.2`；大升级：`2.0`。
*   **修改状态**：版本号格式为 `X.YZ`。
    *   一般只增大 Z 值，X.Y 保持不变（例如：1.13）。
    *   修改完毕变回“正式”时，Z值归零，增加 X.Y 值。

#### 4. 配置项版本管理
目的是按照一定的规则保存配置项的所有版本，避免丢失或混淆，并能快速准确地查找。

#### 5. 配置基线 (Baseline)
*   **定义**：由一组配置项组成，构成一个相对稳定的逻辑实体。
*   **特点**：基线中的配置项被**“冻结”**，不能随意修改。对基线的变更必须遵循正式的变更控制程序。
*   **示例**：产品的一个测试版本（包含需求、设计、代码、测试用例等）。
*   **类型**：
    *   **发行基线**：交付给外部顾客的基线。
    *   **构造基线**：内部开发使用的基线。
*   **定义内容**：建立基线的事件、受控的配置项、建立和变更基线的程序、批准变更所需的权限。

#### 6. 配置管理数据库 (CMDB)
主要内容包括：发布内容、经批准的变更影响、变更请求记录、变更轨迹、特定设备软件、计划升级/替换信息、问题关联等。

#### 7. 配置库
配置库分为三类：

| 类型 | 别名 | 用途与特点 | 权限/管理 |
| :--- | :--- | :--- | :--- |
| **开发库** | 动态库、程序员库、工作库 | 保存当前正在开发的实体（如新模块、文档）。 | 开发人员个人工作区，自行控制，可任意修改。 |
| **受控库** | 主库 | 包含当前的基线以及对基线的变更。 | 被置于完全的配置管理之下，修改需走变更流程。 |
| **产品库** | 静态库、发行库、软件仓库 | 包含已发布使用的各种基线的存档。 | 被置于完全的配置管理之下，作为最终产品，一般不再修改。 |

**建库模式**：
1.  **按配置项类型建库**：适用于通用软件开发（产品继承性强，工具统一）。优点是统一管理，缺点是目录结构可能过于复杂。
2.  **按开发任务建库**：适用于专业软件/定制软件开发（工具繁多，线性发展）。优点是灵活。

![](./images/第19章_配置与变更管理/pic_005.jpg)

### 19.1.2 角色与职责

#### 1. 配置管理负责人 (配置经理)
负责管理和决策整个项目生命周期中的配置活动：
*   管理所有活动（计划、识别、控制、审计）。
*   审批配置库的结构性变更。
*   定义配置管理流程、范围、属性。
*   指派审计员，对成员进行培训。

#### 2. 配置管理员 (CMO)
负责实施具体的配置管理活动：
*   建立和维护配置管理系统、配置库。
*   **配置项识别**。
*   建立和管理基线。
*   版本管理和配置控制。
*   配置状态报告。
*   配置审计、发布管理。

#### 3. 配置项负责人
确保所负责的配置项的准确和真实：
*   记录变更，维护关系。
*   调查审计差异。

![](./images/第19章_配置与变更管理/pic_006.jpg)

### 19.1.3 目标与方针
目标：在整个软件生命周期中建立和维护项目产品的完整性。

### 19.1.4 管理活动
配置管理的日常管理活动主要包括 6 个方面：

1.  **制订配置管理计划**：规定如何开展配置管理，需 CCB 审批。
2.  **配置项标识**：识别配置项，分配标识和版本号，确定命名规则。
3.  **配置项控制**：即对配置项和基线的变更控制。
4.  **配置状态报告**。
5.  **配置审计**。
6.  **配置管理回顾与改进**。

#### 3. 配置项控制 (变更控制)
流程如下：
1.  **变更申请**：提交给 CCB。
2.  **变更评估**：CCB 评估变更的影响、必要性、可行性、工作量。
3.  **通告评估结果**：CCB 通知干系人（批准/否决）。
4.  **变更实施**：项目经理组织修改。
5.  **变更验证与确认**：测试验证，提交 CCB 确认。
6.  **变更的发布**：配置管理员纳入基线，通知相关人员。

![](./images/第19章_配置与变更管理/pic_007.jpg)
![](./images/第19章_配置与变更管理/pic_008.jpg)

**基于配置库的变更控制流程 (示例)**：
1.  将待升级基线从**产品库**取出，放入**受控库**。
2.  程序员从受控库**检出 (Check out)** 代码到**开发库**（代码被锁定）。
3.  修改完成后，**检入 (Check in)** 回受控库（解除锁定）。
4.  全部完成后，新基线存入**产品库**。

![](./images/第19章_配置与变更管理/pic_009.jpg)
*图19-2 基于配置库的变更控制*

#### 4. 配置状态报告
内容包括：受控配置项的标识和状态、变更申请状态、基线状态、其他记录。

#### 5. 配置审计
包括**功能配置审计**和**物理配置审计**。

| 审计类型 | 目的 | 验证内容 |
| :--- | :--- | :--- |
| **功能配置审计** | 验证**一致性**<br>(实际功效是否与需求一致) | 1. 配置项开发已圆满完成。<br>2. 达到规定的性能和功能特征。<br>3. 操作和支持文档已完成并符合要求。 |
| **物理配置审计** | 验证**完整性**<br>(物理存在是否与预期一致) | 1. 要交付的配置项是否存在。<br>2. 是否包含了所有必需的项目。 |

**审计场景**：实施新库后、重大变更前后、发布前、灾难恢复后、发现未授权配置项后等。

![](./images/第19章_配置与变更管理/pic_010.jpg)

#### 6. 配置管理回顾及改进
定期回顾，制定服务改进计划。

## 19.2 变更管理

### 19.2.1 管理基础

*   **实质**：根据项目认知，调整方向和资源，最大程度满足需求。
*   **与配置管理的关系**：配置管理是系统基础，变更管理是针对基准调整的手段。
*   **产生原因**：范围定义疏忽、增值变更、风险应对、被动调整、外部事件等。
*   **分类**：
    *   性质：重大、重要、一般。
    *   迫切性：紧急、非紧急。

![](./images/第19章_配置与变更管理/pic_011.jpg)
![](./images/第19章_配置与变更管理/pic_012.jpg)

### 19.2.2 管理原则
1.  **基准管理**：基准是变更的依据。
2.  **变更控制流程化**。
3.  **明确组织分工**。
4.  **评估变更的可能影响**。
5.  **妥善保存文档**。

![](./images/第19章_配置与变更管理/pic_013.jpg)

### 19.2.3 角色与职责
1.  **项目经理 (PM)**：响应变更，评估影响，实施变更。
2.  **变更管理负责人 (变更经理)**：负责变更过程方案、监控、协调资源、日程安排。
3.  **变更请求者**：提交变更，初步评价风险。
4.  **变更实施者**：执行具体变更任务。
5.  **变更顾问委员会**：对重大变更行使审批或提供建议（紧急变更时由授权者审批）。
6.  **变更控制委员会 (CCB)**：负责批准或否决变更请求。

![](./images/第19章_配置与变更管理/pic_014.jpg)
![](./images/第19章_配置与变更管理/pic_015.jpg)

### 19.2.4 工作程序 (重点)

**标准流程**：
1.  **变更申请**：书面形式提交。
2.  **变更初审**：确认必要性、格式校验、达成共识。
3.  **变更方案论证**：技术可行性与经济论证，将技术要求转化为资源需求。
4.  **变更审查**：CCB 决策（客户和服务对象的意见应放在核心位置）。
5.  **发出通知并实施**：基准调整，资源到位。
6.  **实施监控**。
7.  **效果评估**：评估是否达到目的，与实施过程的差距。
8.  **变更收尾**：判断项目是否纳入正常轨道。

![](./images/第19章_配置与变更管理/pic_016.jpg)
![](./images/第19章_配置与变更管理/pic_017.jpg)
![](./images/第19章_配置与变更管理/pic_018.jpg)

### 19.2.5 变更控制
小项目也要注意规范化：对变更确认应当正式化，操作过程应当规范化。

![](./images/第19章_配置与变更管理/pic_019.jpg)

### 19.2.6 版本发布和回退计划
*   **准备工作**：回退分析、备份（存储过程、配置数据、在线接口等）、确定触发条件。
*   **回退步骤**：通知用户及关联系统 -> 回退数据/程序 -> 通知完成 -> 测试 -> 恢复运行。

## 19.3 项目文档管理

文档分为三类：

| 类型 | 作用 | 示例 |
| :--- | :--- | :--- |
| **开发文档** | 描述开发过程本身 | 可行性研究、需求规格、功能规格、设计规格、测试计划等。 |
| **产品文档** | 描述开发过程的产物 | 培训手册、参考手册、用户指南、支持手册。 |
| **管理文档** | 记录项目管理的信息 | 进度变更记录、职责定义、项目计划、配置管理计划。 |

**文档质量分级**：
1.  **最低限度文档 (1级)**：自用程序（程序清单、开发记录、测试数据、简介）。
2.  **内部文档 (2级)**：专用程序，未共享（含足够注释）。
3.  **工作文档 (3级)**：单位内若干人联合开发，或可被其他单位使用。
4.  **正式文档 (4级)**：正式发行供普遍使用的软件产品。

**规范化管理**：
*   文档书写规范。
*   图表编号规则。
*   文档目录编写标准。
*   文档管理制度（借阅、权限等）。

D. 审计软件即使发现不一致的情况，也不允许自动更新配置库或配置管理数据库，必须由有关负责人调查后再进行更新
